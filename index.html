Uber Data Analysis Project

This study is based entirely on the results of an 8-month period (from May 2017 - December 2017) project in which a simple survey along with the recorded data from an independent contractor’s mobile application (Uber partner/driver app) brought together my study’s main dataset. I personally collected this information by asking the passengers the following questions: “Where are you from originally?” and “What do you like the most about San Diego?”. The intention of this study is to compile, visualize and interpret data in order to learn what aspect of the City of San Diego, California individuals considered to be the most appealing.


The data was compiled in a Microsoft Excel spreadsheet. When the data exploration and cleaninsing phase of my analysis was completed during 2018, the appealing aspects were categorized as the following: W (weather), P (people), O (other) and N/A. In addition, other applications such as Tableau Public and some Microsoft Office tools were used to complete this study.

Only data from columns A, B, D, E, G, H, and I, were extracted from the Uber Driver Mobile App. Columns C, F and J,
contain information obtained from the author. Column K represents the survey's results.
The final dataset was exported in Tableau Public and results demonstrated that most passengers (which represent a
random population sample) consider that the weather is San Diego's best attribute.

In this project we were not only able to observe a pattern regarding the city's favorable weather, this study also allowed the author the opportunity to observe other trends such as the passenger's drop off zones' zip codes or the most visited neighborhoods based on the riders' demand.

This image was generated using Tableau Public and represents every drop-off
zone or zip code visited during the data exploration phase of the project.
The chart demonstrates that UberX service or "private trips" are more common
than UberPOOL rides. POOL service started in June 2017, that is why May has
zero rides throughout UberPOOL.
Downtown San Diego is the passengers' favorite destination and SAN Airport
despite not being a neighborhood itself and a specific location, is a common
drop-off location as well.

Text File Comparer Automation Project

The author came up with the idea after observing that data delivered on a quarterly basis followed identical trends. One out of the several tasks as a Data Analyst was to compare several text files by manually manipulating data on Microsoft Excel (spreadsheet) prior to updating a mobile application known as Market Metrics. At the beginning of each quarter, some clients would deliver a list of their ‘Broker Dealer Exclusion IDs’ or ‘BDE IDs’ that were related to certain data/transactions that each client did not want to see in their account linked with the mobile app. As part of the process, these lists (or txt files) containing the BDE IDs were compared to those lists submitted by each iPad client during the previous quarter. This was manually done by aligning each column of data and inserting a 3rd column with boolean datatype (True or False).

The first step was to obtain the text files from the current and previous quarter.
Each row of data from the text file gets copied and pasted into an Excel spreadsheet.
Then the Analyst would typically copy and paste several text files into a spreadsheet and drag
a boolean formula in order to make sure that the Exclusion IDs match, and if not, then figure
out why.

This process may not seem too difficult and it may not require advanced knowledge in Microsoft Excel, however, it is time consuming and unnecessary. There is no denying that most technical processes are still required to be performed by humans, but it is both more time and resource efficient to have a Python script or code that can handle copying and pasting large amounts of data from different paths (with several folders within), and display the results in a matter of seconds!

Once the code starts running, it will prompt the user/analyst to enter the name of his/her client and the current quarter, by default the code will
automatically pick the select the previous quarter.
The code will seek through the folder paths depending on the user's input and will extract data from each corresponding text file.

The script takes the data from the text files and stores it into lists; then it will create a 3rd list containing boolean datatype once the current
and previous quarter lists get compared. The three lists get aligned with each other and results get printed out at the end.

The section of the image being point at by the red arrow represents the final result after the automation was finally completed. The image to the left represents a spreadsheet completed manually.

Data Processing and Database Project

The purpose of this technical project is to describe the steps required to obtain raw data from a source (in this case a digital thermometer) and convert the information throughout a series of steps into quality, presentable and understandable data available for Microsoft Excel which eventually gets stored into a database.

The Arduino application is open-source. The user can program the board
before wiring the components. Writing the code is the first step.
This diagram represents the way in which the Arduino UNO board must get
wired, so that the circuit can mimic a digital thermometer. The diagram can be
modified depending on how the user decides how the UNO board will input
electricity into the circuit. Connecting or wiring UNO is the second step.
This is a picture of an operational Arduino thermometer; it displays results in
Celsius and Fahrenheit. Code gets loaded into the circuit once this is connected
via USB. The third step is to verify that results are being displayed correctly.

After the thermometer was connected and working accordingly, a python script linking the author's computer and the device, was created out of scratch. The function of this code is to extract real-time data, link each data point to a timestamp and convert each point into a row that gets pasted in a plain text file. The platform used for creating the python code was PyCharm CE.

The fourth step was to create the code that could compile the data from the thermometer. As the python code
runs, data is getting inserted into a plain text file.
As you may notice in this picture, each row of data consists of the following: degrees Celsius, degrees
Fahrenheit, date (in MM/DD/YYYY format) and time (in HH/MM/SS format).

After the python script completes it's function, an Excel macro takes care of processing and arranging the data accordingly. It is not possible to load the data directly from the raw data file since this got imported into a text file with commas. In order for the data to meet the database's format requirements, the components need to get split into several columns with the appropriate format.

The fifth step was to record a macro since importing the .txt file and processing each column into an Excel spreadsheet
consisted of performing repetitive steps.
This is the final result. Once the Macro processed the data, the spreadsheet or Excel file is ready to get loaded into the
database.

The final step was to create a small relational database using SQL. This database was designed to accept decimal numbers corresponding to temperature in Celcius and Fahrenheit; and both time and date in the appropriate format. The last condition that data must meet is that rows cannot have any null values.

The sixth step was to create the SQL database adjusted to accept raw real time data from the thermometer only.
Once that step is completed data can get loaded and it becomes viewable. However this can be see by writing a
query that selects all data from every column that corresponds to the recorded room temperature.

Each step along the process required knowledge in C/C++, Python, Microsoft Excel - Macro, Microsoft Excel - VBA, and SQL. Please recall that to get information loaded into a database created with SQL, information requires to be formatted appropriately to assure that it gets stored with no errors and to mantain the quality of the data.

Three-D Computer Aided Design (CAD) Designs

The following section will include images of some objects that were designed by the author with solid modeling software. Modern mechanical and aerospace engineers, industrial designers, technical draftsman and even hobbyists use this type of applications to create objects that are commonly seen in our daily lives. Although there are many applications that serve the same purpose, the only ones that have been used by the author are the following: Creo Parametric, SolidWorks and Fussion 360. By definition, isometric view is when the drawing of an object has all of its lines set to have a 30 degree angle with respect to their axis; meaning that a drawing of an object is slightly shifted so that it could look like a 3D object instead of a 2D drawing. Exploded view refers to an assembly (group of parts that forms an object/machine/tool) that has all of it's components separated at a distance relative to each other or to its origin an xyz cartesian system.

The Bio-inspired Glucose Detector

This was our final senior project for an Optics course and it made us physically interpret what the given Matlab code was displaying. Our job was to modify some aspects of the code and understand the way light traveled throughout a photonic crystal, so we could hypothetically invent a sensor that works by using these principles to measure the amount of glucose contained within any substance. We came up with the idea of a small device that had its surface covered with aluminum-oxide barbs (or tiny cylinders like ones from the image below). The picture represents the microscopic structure of a Peacock's feather. Unfortunately, we don't have an image of the hypothetical device, however, we concluded that the greater the angle in which light got bend in presence of the sensor's barbs, implied that the more glucose was contained in whatever substance (or solution).
