Uber Data Analysis Project

This study is based entirely on the results of an 8-month (from May 2017 - December 2017) project in which a simple survey along with the recorded data from an independent contractor’s mobile application (Uber partner/driver app) brought together my study’s main dataset. I personally collected this information by asking the passengers the following questions: “Where are you from originally?” and “What do you like the most about San Diego?”. The intention of this study is to compile, visualize and interpret data in order to learn what aspect of the City of San Diego individuals consider to be the most appealing.


The data was compiled in a Microsoft Excel spreadsheet. When both, the data exploration and cleansing phases of my analysis were completed during 2018, the appealing aspects were categorized as the following: W (weather), P (people), O (other) and N/A. In addition, other applications such as Tableau Public and some Microsoft Office tools were used to complete this study.

A) Only data from columns A, B, D, E, G, H, and I, were extracted from the Uber Driver Mobile App. Columns C, F and J,
contain information obtained from the author. Column K represents the survey's results.
B) The final dataset was exported in Tableau Public and results demonstrated that most passengers (which represent a
random population sample) consider that the weather is San Diego's best attribute.

In this project we were not only able to observe a pattern about the riders' opinion regarding the city's favorable weather. This study also allowed me the opportunity to observe other trends such as the passenger's drop off zones' zip codes or the most visited neighborhoods based on the riders' demands.

This image was generated using Tableau Public and represents every drop-off
zone or zip code visited during the data exploration phase of the project.
The chart demonstrates that UberX service or "private trips" are more common
than UberPOOL rides. POOL service started in June 2017, which is why May has
zero rides throughout UberPOOL.
Downtown San Diego is the passengers' favorite destination. SAN Airport
despite not being a neighborhood itself but a specific location with the same
zip code as Downtown, is a common drop-off location as well.

Text File Comparer Automation Project

This idea was developed after observing that data delivered on a quarterly basis followed identical trends. One of the tasks as a Data Analyst, was to compare several text files by manually manipulating data on Microsoft Excel (spreadsheet) prior to updating a mobile application known as Market Metrics. At the beginning of each quarter, some clients would deliver a list of their ‘Broker Dealer Exclusion IDs’ or ‘BDE IDs’. These IDs are directly linked to certain transactions that ocurred during the quarter. Each client had the benefit to select which Broker Dealers were going to get excluded on a quarterly basis. Once exclusions were applied, the client could not see (through the mobile app) those transactions linked with the BDE ID's. As part of the process, these lists (or txt files) containing the BDE IDs were compared to those lists submitted by each iPad client during the previous quarter. This was manually done by aligning each column of data and inserting a 3rd column with boolean datatype (True or False).

A) The first step was to obtain the text files from the current and previous quarter.
B) Each row of data from the text file gets copied and pasted into an Excel spreadsheet.
C) The Analyst would typically copy and paste several text files into a spreadsheet and drag
a boolean formula in order to make sure that the Exclusion IDs match. Ideally the BDE ID's
should be the same every quarter, however this is not always the case. ID's might not match
because clients may select distinct broker dealers that need to get excluded.

This process may not seem too difficult and it may not require advanced knowledge in Microsoft Excel, however, it is time consuming and unnecessary. There is no denying that most technical processes are still required to be performed by humans, but it is both more time and resource efficient to have a Python script or code that can handle copying and pasting large amounts of data from different paths (with several folders within), and display the results in a matter of seconds!

A) Once the code starts running, it prompts the user/analyst to enter the name of his/her client and the current quarter. By default, the code
automatically selects the previous quarter as well.
B) The code seeks through the folder paths depending on the user's input, and extracts data from each corresponding text file.

C) The script takes the data from the text files and stores it into lists; then it creates a 3rd list containing boolean datatype once the current
and previous quarter lists get compared. The three lists get aligned with each other and results get printed out at the end. The image to the right
represents the final step prior to displaying the three lists (including the column with Boolean datatype results).

D) The image to the left represents a spreadsheet completed manually.The section of the image pointed by the red arrow represents the final result after the automation process is finally complete.

Data Processing and Database Project

The purpose of this technical project is to describe the steps required to obtain raw data from a source (in this case a digital thermometer). Information then gets converted throughout a series of steps, into presentable and understandable data available for Microsoft Excel. Once the data gets processed, temperature information is ready to get stored into a database.

A) The Arduino application is open-source. The user must program the board
before wiring the components. Writing the code is the first step.
B) This diagram represents the way in which the Arduino UNO board must get
wired, so that the circuit can mimic a digital thermometer. The diagram can be
modified depending on how the user decides to input electricity from UNO
into the circuit. Connecting or wiring UNO is the second step.
C) This is a picture of an operational Arduino thermometer; it displays results in
Celsius and Fahrenheit. Code gets loaded into the circuit once this is connected
via USB. The third step is to verify that temperature is being displayed correctly.

The thermometer/Arduino needs to get connected to a computer in order to run, however, a Python script was created out of scratch with the purpose of linking the author's computer and receiving data from the device. The function of this code is to record real-time data, link each data point to a timestamp and convert each point into a row that gets pasted in a plain text file. The platform used for creating the Python code was PyCharm CE.

D) The fourth step was to create the code that could compile the data from the thermometer. As the Python code
runs, data gets inserted into a plain text file immediately.
E) As you may notice in this picture, each row of data consists of the following: degrees Celsius, degrees
Fahrenheit, date (in MM/DD/YYYY format) and time (in HH/MM/SS format).

After the python script completes its function, an Excel macro takes care of processing and arranging the data accordingly. It is not possible to load the data directly from the raw data file since it gets imported into a text file with commas. In order for the data to meet the database's format requirements, the components need to get split into several columns with the appropriate format.

F) The fifth step is to record a macro since importing the .txt file and processing each column into an Excel spreadsheet
consists of performing repetitive steps.
G) This is the final result. Once the Macro processes the data, the spreadsheet or Excel file is ready to get loaded into the
database.

The final step is to create a small relational database using SQL. This database is designed to accept decimal numbers corresponding to temperature in Celcius and Fahrenheit; and both time and date in the appropriate format. The last condition that data must meet is that rows cannot have any null values.

H) The sixth step is to create the SQL database adjusted to accept processed or modified data recorded from
the thermometer only. Once that step is complete, data can get loaded and it becomes viewable. Data can
be seen by writing a query that selects all data from every column that corresponds to the recorded room temperature.

Each step along the process required knowledge in C/C++, Python, Microsoft Excel - Macro, Microsoft Excel - VBA, and SQL. Please recall that to get information loaded into a database created with SQL, data requires to be formatted appropriately to assure that it gets stored with no errors.

Three-D Computer Aided Design (CAD) Designs

The following section contains images of some mechanical devices that were designed by myself with solid modeling software. Modern mechanical and aerospace engineers, industrial designers, technical draftsman and even hobbyists use these types of applications to create objects that are commonly seen in our daily lives. I currently have experience with the following: Creo Parametric, SolidWorks and Fussion 360. By definition, isometric view is when the drawing of an object has all of its lines set to have a 30 degree angle with respect to their axis; meaning that a drawing of an object is slightly shifted so that it looks like a 3D object instead of a 2D drawing. Exploded view refers to an assembly (group of parts that form an object/machine/tool) that has all of its components separated at a distance relative to each other or to its origin in the xyz-cartesian system.

The Bio-inspired Glucose Detector

This was our final senior project for an Optics course at San Diego State University. The objective was to physically interpret what the given Matlab code was displaying. The next step was to modify some aspects of the code in order to understand the way light traveled throughout a photonic crystal. Our goal was to design a sensor. This sensor or device contains the modified Matlab script embeddeb within, with the goal of measuring the amount of glucose contained in any substance. We came up with the idea of a small device that had its surface covered with aluminum-oxide barbs. The sensor is similar to the microscopic structure of a Peacock's feather (which contains photonic crystals), that refracts light as the angle and position of the feather changes. Unfortunately, we don't have an image of the hypothetical device, however, we concluded that the greater the angle in which light got bent in presence of the sensor's barbs, the more glucose was contained in whatever substance (or solution).
